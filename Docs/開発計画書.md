## 2. フェーズ別サマリー

### Phase 0 — 準備・キックオフ

- 体制・コミュニケーション（定例、意思決定者）
- リポジトリ・ブランチ方針・環境（開発 / ステージング）
- 仕様書の共有と、§12 未決事項の優先づけ

**完了条件（例）:** キックオフ完了、Phase 1 の入力が揃っている

---

### Phase 1 — 詳細設計

- 画面一覧・主要フロー、ワイヤーまたはモック
- ER 図・テーブル定義案、RLS の方針
- ロール（本部 / 現場等）と操作権限マトリクス
- 仕様書 §12 のうち実装前に閉じる項目の決定

**完了条件（例）:** 設計レビュー承認、Phase 2 の実装タスクに分解済み

---

### Phase 2 — データ基盤・認証

- Supabase（または採用DB）のスキーマ反映、マイグレーション
- RLS（社内のみ・ロールに応じた読み書き）
- アプリからの認証フロー実装

**完了条件（例）:** ステージングで RLS を含む疎通確認済み

---

### Phase 3 — 説明会コア

- 説明会の登録・編集・参照・一覧・検索（仕様書 §6.2, §6.7）
- 管理単位は **説明会のみ**；工事名等は属性

**完了条件（例）:** 説明会のみでデモ可能（他エンティティは未実装でも可）

---

### Phase 4 — 配布資料・版管理

- 資料の添付またはリンク、版・更新者・日時（§6.3）
- ストレージ容量・拡張子制限など運用ルールの実装

**完了条件（例）:** 1 説明会に複数資料・版の追跡が可能

---

### Phase 5 — Q&A・出席・決定事項

- Q&A（§6.4）、出席は **世帯数のみ**（§6.5）、決定事項と参照（§6.6）

**完了条件（例）:** 仕様書 MVP 機能要件（§6）をすべて満たす UI/API

---

### Phase 6 — 結合試験・社内パイロット・本番

- 結合・回帰テスト、セキュリティ観点の確認
- 社内パイロット、フィードバック反映
- 本番デプロイ、運用手順・教育、KPI 測定の開始（§9）

**完了条件（例）:** 本番利用開始、Phase 6 進捗報告にリリースノートを添付

## 4. 依存関係

```text
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6
                              └─────────┴─────────┴─ 順次拡張（3→5）
```

- Phase 3〜5 は同一アプリ内の機能のため、**ブランチ戦略**（main への段階マージ or feature flag）はチームで決定する。
- Phase 4・5 は Phase 3 の「説明会」エンティティに依存する。

## 5. 成果物チェックリスト（MVP）

仕様書 [§6 機能要件（MVP）](./開発仕様書_第1版_住民説明まわりMVP.md#6-機能要件mvp) に対応:

- [ ] 説明会の登録・編集・一覧・検索
- [ ] 配布資料（版・更新者・日時）
- [ ] Q&A
- [ ] 出席（世帯数のみ）
- [ ] 決定事項と参照
- [ ] 社内利用のみ（外部ポータルなし）
- [ ] 権限に応じたアクセス制御
