## 2. エグゼクティブサマリー

### 2.1 背景

解体工事を行う建設業の施工本部において、業務手順と課題を整理した資料（欲望リスト）を踏まえ、特に**住民説明・説明会**に関する準備・実施・記録の属人化や手戻りを抑える必要がある。

### 2.2 目的

- 説明会に関する**資料・Q&A・決定事項**を一箇所に集約し、担当交代でも業務を継続できるようにする。
- **誰が・いつ・何を配布・回答したか**を追えるようにし、再説明やトラブル時の説明負荷を下げる。
- 経営が追認可能な形で、**投資対効果の測定（KPI）**の土台を用意する。

### 2.3 投資対効果の見立て（定性）

- 説明会の**資料差し替え・再開催**に伴う工数・機会損失の抑制。
- Q&A・決定事項の散在による**認識齟齬・再作業**の抑制。

## 3. スコープ

### 3.1 対象業務（「説明まわり」狭義）

次を MVP の対象とする。

- 説明会・住民説明の**企画・日程・実施管理**
- **配布資料**の作成・更新・版管理
- **質疑応答（Q&A）**の整理・回答案・共有
- **出席**の記録（**世帯数のみ**。氏名は扱わない）
- **決定事項**の記録と、関係者（社内）への参照しやすさ

### 3.2 MVP の範囲（IN）

| 区分 | 内容 |
|------|------|
| 説明会 | 1回の説明会を**管理の主単位**とする（複数回説明会を束ねる親階層は MVP では設けない）。工事名・現場名等は**属性**として紐づけ可能とする。 |
| 資料 | アップロードまたはリンク、版、更新者・日時 |
| Q&A | 質問の登録、回答案、確定回答、社内での共有 |
| 出席 | **世帯数のみ**（個人を特定する情報は登録しない） |
| 決定事項 | 文面、日付、関連資料への参照 |

### 3.3 対象外（OUT）・フェーズ2以降

- 説明会以外を主目的とする**近隣窓口・苦情管理**の全面システム化（広義の「近隣対応」）。
- **行政・警察向け説明**を、住民説明と同一機能ですべてカバーすること（必要なら別要件）。
- 欲望リストに含まれる**工程表・搬入ルート・AR/CAD 連携**等、説明会狭義と直接結びつかない機能。
- **`source/欲望リスト（雛形）.xlsx` シート2**に関する要件（別フォーマットで入力予定のため、本版では記載しない）。

## 4. ステークホルダーと利用者

| ステークホルダー | 役割の例 |
|------------------|----------|
| 経営 | 方針・投資・KPI の承認 |
| 施工本部 | 説明会の企画・調整・資料・Q&A・記録の責任 |

※ MVP は**社内利用のみ**のため、住民・協力業者をシステムの利用者として想定しない（外部ポータルは作らない）。

## 5. 業務要件（As-Is / To-Be）

### 5.1 現状の流れ（例）

- 資料・Q&A・決定事項が **Excel・メール・紙**等に分散し、版管理が難しい。
- 担当者依存で、引き継ぎ時に**文脈が失われる**。

### 5.2 課題（例）

- 再説明・差し替えのたびに**同じ調整を繰り返す**。
- 「いつその回答になったか」「どの版の資料か」が追いにくい。

### 5.3 目指す流れ（To-Be）

- **説明会単位**で、資料・Q&A・決定事項・出席（世帯数）を一箇所に集約。
- 社内の権限の範囲で**閲覧・編集・履歴**が追える。

## 6. 機能要件（MVP）

### 6.1 管理単位・データの考え方（確定事項）

| 項目 | 決定 |
|------|------|
| 1件あたりの単位 | **説明会のみ**（複数回説明会を束ねる親エンティティは MVP では作らない） |
| 出席 | **世帯数のみ**（氏名・住所等の個人情報は登録しない） |
| 利用範囲 | **社内のみ**（住民・業者向けの公開ポータルは MVP に含めない） |

### 6.2 説明会（イベント）管理

- 説明会の登録・編集・一覧
- 属性例：タイトル、日時、場所、担当、ステータス（予定／実施済／中止 等）、**工事／現場の識別子または名称**（文字列または外部マスタ連携は実装段階で決定）

### 6.3 配布資料（版管理・更新履歴）

- ファイルの添付または URL
- 版番号または更新日時の表示
- 更新者・更新日時の記録

### 6.4 質疑応答（Q&A）管理

- 質問の登録、回答案、確定回答
- 説明会への紐づけ
- 状態（未回答／回答確定 等）は実装時に詳細化

### 6.5 出席記録

- **世帯数**の数値入力（または集計値の入力）
- 個人を特定する情報は扱わない

### 6.6 決定事項の記録・参照

- 決定事項テキスト、決定日、関連する資料・Q&A への参照（リンク）

### 6.7 検索・一覧（本部向け）

- 説明会一覧、条件検索（日付・担当・ステータス等）

## 7. 非機能要件

### 7.1 セキュリティ・アクセス制御

- 社内ユーザーのみ利用可能とする（認証方式は実装設計で決定）。
- ロールに応じた**閲覧・編集**の制限（詳細は実装設計で定義）。

### 7.2 個人情報・プライバシー

- MVP では出席を**世帯数のみ**とし、住民の氏名・住所・連絡先等は原則保持しない。
- 資料に個人情報が含まれる場合の取り扱いは**社内規程**に従い、運用で定める。

### 7.3 可用性・バックアップ

- 利用時間帯・バックアップ方針はインフラ選定時に定義（本書では方針レベル）。

### 7.4 監査・ログ

- 重要な変更（資料更新、決定事項の変更等）の**履歴が追えること**を推奨。必須範囲は実装設計で確定。

## 9. 成功指標（KPI）

以下は**定義と測定方法を運用開始後に確定**する想定の候補。

- 同一現場・同一説明テーマにおける**資料の全面差し替え回数**／説明会あたり
- Q&A の**未回答滞留日数**
- 説明準備に関する**記録ベース工数**（取得できる場合）

---

## 10. リスクと対応方針

| リスク | 対応方針（例） |
|--------|----------------|
| 運用が定着せず二重管理になる | 本部と現場で**必須入力項目**を最小化し、運用ルールを明文化 |
| 資料に個人情報が混入 | アップロードガイドラインと教育、マスキング運用 |
| スコープ肥大（広義の近隣対応まで拡大） | 本書の OUT を維持し、変更は**改訂版**で承認 |
